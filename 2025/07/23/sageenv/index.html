  1. (Basic) Use Sagemath in VS Code, with Syntax Parsing and other helps.

Run sage ×; Import sage √

Sagemath supports many special syntaxes for simplicity and convenience. Take the following code as an example, you can use it directly in a console or write it to a main.sage and run it with sage main.sage. Typically, it is least efficient to run code in a console.

But, as we all know, Sagemath is essentially python-style code, and we can use it exactly as python. In this case, you can import sage like a package and run the code with Sagemath’s built-in python interpreter as sage -python main.py.

What are the benefits of this?

As mentioned before, Sagemath supports its own syntax system. When implementing python code, the meaning of some code is implicitly changed. For example, sagemath will use its own type by default for numbers. Another common example is the ^ symbol in sagemath, which is equivalent to ** in python. However, ^ is used in python for xor computation, for which you will have to use ^^ in sagemath instead.

In this case, running your code (especially external python packages or projects) directly with Sagemath can easily have an obscure effect on type identification, function calls, etc. This is intolerable for me, so I highly recommend the python-style solution.

I can’t fully explain how Sagemath works, but this works perfectly well: for any data type, function, syntax in Sagemath, you can always find the corresponding python code to implement it. In fact, you can always find a main.sage.py after you run sage main.sage, which contains the python-version of your Sagemath codes.

For efficiency, I don’t think there is a significant difference between the two ways. If you wanna run sage-independent python code, using the python interpreter directly is a more efficient choice.

Another advantage is that if you implenemt your code entirely in python style, you can introduce python-related aids such as syntax highlighting, which we’ll cover later.

A Fresh Take

Through further experimentation, I discovered a more interesting method. You see, we can use .../bin/sage-python mentioned above as a python interpreter, can we set up a virtual environment using venv? The answer is yes, but it’s just a pure python environment (without sage package).

But it turns out that we can import sage by inheriting the system package, as .../bin/sage-python -m venv my_new_sage_venv --system-site-packages. In this way, all virtual environments share the same Sagemath base environment, but you can make special configurations in them.

The advantages are obvious: pure, independent virtual python environment, which makes configuration much easier. We donnot need additional configurations as described earlier, we just need to configure the current virtual environment in VS Code. And we can run our code as python main.py. And also we can use this python interpreter for Jupyter Notebook.
